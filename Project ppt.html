Project Overview & Innovations

Project Description

This project implements a robust system for classifying product reviews as Positive, Negative, or Neutral using a fine-tuned DistilBERT model. The end-to-end pipeline covers data ingestion, ETL, model training, and deployment of a sentiment scoring API using FastAPI and Docker.

Key Innovations

  • GenAI-Powered Chatbot with RAG: An empathetic, context-aware chatbot using Google's Gemini API.
  • Innovative Weighted Loss Training: A custom trainer that prioritizes "helpful" reviews to maximize accuracy.
  • Model Interpretability with LIME: A transparency dashboard to explain the "why" behind AI predictions.
  • Scalable & Efficient Data Pipeline: A server-friendly pipeline for processing millions of reviews on standard hardware.
  • Rigorous Hypothesis-Driven Experimentation: A data-validated approach to find the optimal training strategy.

The Dataset: Amazon Product Reviews

Source & Format

The project uses the large-scale Amazon Product Reviews dataset, provided as numerous JSONL (JSON Lines) files.

Key Fields Used:

  • `overall`: The 1-5 star rating, used for sentiment labeling.
  • `helpful`: Community votes, used for the weighted loss function.
  • `reviewText` & `summary`: Combined to form the input text for the model.

The Data Pipeline: From Raw to Ready

1. Parallel Ingestion

Python's `multiprocessing` library is used to read and process multiple JSONL files concurrently, drastically reducing I/O time.

2. Memory-Efficient ETL

Leverages the Hugging Face `datasets` library to handle massive files without loading them entirely into RAM, preventing memory overflow.

  • Cleaning: Removes residual HTML tags from review text.
  • Feature Engineering: Concatenates `title` and `reviewText` for richer model input.

1. Sentiment Analysis

DistilBERT is a lighter and faster version of Google's BERT language model, chosen for its efficiency. In this project, it was specifically fine-tuned using an innovative "Weighted Loss" strategy, which trained the model to pay more attention to reviews the community had already marked as "helpful". Ultimately, this custom-trained model decisively outperformed the baseline and other methods, achieving 88.14% accuracy, and was saved as the final, production-ready model for the application.

2. Model Transparency (LIME)

LIME: Local Interpretable Model-agnostic Explanations (LIME) is a technique that explains why a complex "black box" AI model made a specific prediction on a single piece of data. It works by creating small variations of the input‚Äîlike removing words from a review‚Äîand observing how those changes affect the model's final decision. By doing this, LIME identifies and highlights the key words that most influenced the outcome, which helps users understand and trust the AI's reasoning.

3. GenAI Chatbot (RAG)

RAG: It's considered a Retrieval-Augmented Generation (RAG) system because instead of retrieving factual text, it first retrieves the user's emotional state using its own specialized sentiment model. This retrieved sentiment then augments the prompt sent to the Gemini AI, providing critical context. Finally, Gemini generates a response that is intelligently tailored to the user's detected emotion, making it empathetic and context-aware.

Challenges & Solutions

Challenge: Handling Massive Dataset on Limited Hardware

The Amazon reviews dataset was too large to fit into memory, causing crashes. We solved this by using the Hugging Face `datasets` library for memory-efficient processing and Python's `multiprocessing` to speed up data ingestion.

Challenge: Ensuring Fair Model Evaluation with Imbalanced Data

The dataset had a strong bias towards positive reviews, which could lead to a model that performs poorly on negative or neutral sentiment. We addressed this by implementing a stratified sampling strategy to create a balanced dataset, ensuring the model was trained and evaluated on an equal representation of all sentiment classes.

Challenge: Creating a Lean and Reproducible Docker Environment

Our initial Docker images were unnecessarily large and slow to build because `pip freeze` included many libraries from the local development environment that were not actually needed for the application. To solve this, we manually audited our code to identify the specific, essential packages. We then created a minimal `requirements.txt` file from scratch. This resulted in a significantly smaller, faster, and more secure Docker image, which is a best practice for production deployments.
